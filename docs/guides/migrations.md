# Database Migrations

Migrations live in `supabase/migrations/` and are managed by the Supabase CLI.

## Commands

```bash
npm run sb:migrate   # apply pending migrations to local DB
npm run sb:reset     # destructive: drop DB + re-apply all migrations from scratch
```

Use `sb:migrate` for incremental updates. Use `sb:reset` when you need a clean slate (e.g. after a conflicting migration or during initial setup).

## Creating a Migration

```bash
supabase migration new <description>
# e.g. supabase migration new add_notes_column
```

This creates `supabase/migrations/YYYYMMDDHHMMSS_add_notes_column.sql`.

Edit the file to add your SQL, then apply it:

```bash
npm run sb:migrate
```

## Naming Convention

Files follow the pattern: `YYYYMMDDHHMMSS_description.sql`

Example: `20240315120000_add_for_trade_column.sql`

The timestamp is automatically generated by the Supabase CLI.

## RLS Pattern

All tables in this project use Row-Level Security. The standard pattern:

```sql
-- Enable RLS
alter table public.cards enable row level security;

-- Policy: users can only access their own rows
create policy "Users can manage their own cards"
  on public.cards
  for all
  using (auth.uid() = owner_id)
  with check (auth.uid() = owner_id);
```

## Important Constraints

- **Do not rename `owner_id`** without also updating `src/lib/supabase/collection.ts`, where all queries filter on `.eq('owner_id', userId)`.
- **Do not rename the `cards` table** without updating all references in `collection.ts`.
- Migrations run in timestamp order — ensure new migrations don't conflict with existing ones.

## Checking Current State

```bash
npm run sb:studio    # open Supabase Studio → Table Editor
npm run sb:status    # show local instance status
```
