> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Fermer les trous de sécurité réels de wizcard.xyz en prod : fuite `purchase_price`, headers HTTP manquants, route proxy Scryfall non validée — sans ajouter de mécanisme redondant avec Supabase.

**Architecture:** Une migration SQL idempotente ferme la fuite `purchase_price` (column-level REVOKE sur `anon` + drop de la policy publique redondante). La route `scryfall/cards/collection` valide la forme et la taille du body avant tout fetch sortant.

**Tech Stack:** Next.js 16 (App Router), Supabase (Postgres + PostgREST, RLS), TypeScript. Migrations SQL dans `supabase/migrations/`. Pas de framework de test.

## Task 1: Migration — fix fuite `purchase_price`

**Files:**

- Create: `supabase/migrations/20260710120000_fix_purchase_price_leak.sql`

**Interfaces:**

- Consumes: table `public.cards`, policies existantes de `20260616000000_public_read_sharing.sql`, view `public.public_collection_cards`.
- Produces: après migration, `anon` ne peut plus lire la colonne `purchase_price` sur `cards` ; la policy `"Public can view collection cards"` n'existe plus.

**Contexte pour l'implémenteur (zéro contexte supposé) :**
La table `public.cards` a une colonne `purchase_price text` (prix d'achat, donnée financière perso). La migration `20260616000000` a créé deux policies publiques :

- `"Public can view deck cards"` `USING (deck_id is not null)` — **à conserver**, elle rend les cartes de deck publiquement lisibles (les decks sont publics).
- `"Public can view collection cards"` `USING (owner_id is not null)` — **le bug** : elle ouvre TOUTE la table (prix inclus) en lecture anonyme, court-circuitant la view `public_collection_cards` qui masque le prix.
  La page publique `/decks/[id]` lit les cartes via `from('cards').select('*')` en client anonyme. Un `REVOKE` global casserait ça. On révoque donc UNIQUEMENT la colonne `purchase_price` pour `anon` : `select('*')` continue de marcher (PostgREST renvoie les colonnes accordées sans erreur), mais le prix disparaît de la réponse. L'owner lit son prix en tant que `authenticated` (RLS ligne), inchangé.

- [ ] **Step 1: Écrire la migration**

Créer `supabase/migrations/20260710120000_fix_purchase_price_leak.sql` :

```sql
-- Fix: purchase_price leaked to anonymous visitors.
--
-- The "Public can view collection cards" policy (USING owner_id is not null)
-- opened the entire public.cards table — including purchase_price — to anon,
-- bypassing the purchase_price-free public_collection_cards view. Deck cards can
-- also carry purchase_price and are read by anon via the deck-cards policy with
-- select('*'), so dropping the policy alone is insufficient.
--
-- Fix: drop the redundant collection policy AND revoke SELECT on just the
-- purchase_price column for anon. select('*') keeps working (PostgREST returns
-- only granted columns); the price is unreadable to anon on every row. The owner
-- reads it as `authenticated` via the existing owner RLS policy (unchanged).
--
-- Idempotent + reversible.

-- 1. Remove the over-broad public collection read (kept the deck-cards one).
drop policy if exists "Public can view collection cards" on public.cards;

-- 2. Column-level revoke: anon can read every column of cards EXCEPT
--    purchase_price. Deck viewing (from('cards').select('*')) is unaffected.
revoke select (purchase_price) on public.cards from anon;
```

## Task 4: Durcir la route proxy `scryfall/cards/collection`

**Files:**

- Modify: `src/app/api/scryfall/cards/collection/route.ts`

**Interfaces:**

- Consumes: le body POST `{ identifiers: [...] }` envoyé par `src/lib/scryfall/endpoints/cards.ts::scryfallPost('/cards/collection', { identifiers })`.
- Produces: la route rejette (`400`/`413`) tout body malformé ou trop gros AVANT le fetch sortant ; les appels légitimes (≤ 75 identifiers) passent inchangés.

**Contexte pour l'implémenteur :**
La route actuelle (`route.ts`) fait `body = await req.json()` puis forwarde le body brut à `https://api.scryfall.com/cards/collection` sans validation — proxy ouvert, utilisable en relais. L'API Scryfall cape à **75 identifiers par requête**, et l'import envoie précisément des batchs de 75 (`scryfall-resolver.ts`, `BATCH_SIZE = 75`). On valide donc : body = objet avec `identifiers` = tableau non vide de longueur ≤ 75, chaque entrée étant un objet. Toute déviation → `400`. Body trop volumineux → `413`. **Pas de rate limit** (décision spec).

- [ ] **Step 1: Réécrire le handler avec validation**

Remplacer entièrement le contenu de `src/app/api/scryfall/cards/collection/route.ts` par :

```ts
import { NextResponse } from 'next/server';

const SCRYFALL_URL = 'https://api.scryfall.com/cards/collection';

// Scryfall caps /cards/collection at 75 identifiers per request; the importer
// already batches by 75 (scryfall-resolver BATCH_SIZE). Anything larger is
// either a bug or abuse of this open proxy — reject before any outbound fetch.
const MAX_IDENTIFIERS = 75;
// Generous cap on raw body size (a 75-identifier batch is a few KB).
const MAX_BODY_BYTES = 64 * 1024;

export async function POST(req: Request) {
	// Reject oversized bodies up front (defends the open proxy from abuse).
	const contentLength = Number(req.headers.get('content-length') ?? '0');
	if (contentLength > MAX_BODY_BYTES) {
		return NextResponse.json({ error: 'Request body too large' }, { status: 413 });
	}

	const raw = await req.text();
	if (raw.length > MAX_BODY_BYTES) {
		return NextResponse.json({ error: 'Request body too large' }, { status: 413 });
	}

	let body: unknown;
	try {
		body = JSON.parse(raw);
	} catch {
		return NextResponse.json({ error: 'Invalid JSON body' }, { status: 400 });
	}

	// Validate shape: { identifiers: object[] } with 1..MAX_IDENTIFIERS entries.
	if (typeof body !== 'object' || body === null || !('identifiers' in body)) {
		return NextResponse.json({ error: 'Missing identifiers' }, { status: 400 });
	}
	const { identifiers } = body as { identifiers: unknown };
	if (
		!Array.isArray(identifiers) ||
		identifiers.length === 0 ||
		identifiers.length > MAX_IDENTIFIERS ||
		!identifiers.every((i) => typeof i === 'object' && i !== null)
	) {
		return NextResponse.json({ error: 'Invalid identifiers' }, { status: 400 });
	}

	const res = await fetch(SCRYFALL_URL, {
		method: 'POST',
		headers: {
			'Content-Type': 'application/json',
			Accept: 'application/json',
			'User-Agent': 'Wizcard/1.0',
		},
		body: JSON.stringify({ identifiers }),
	});

	const data = await res.json();

	if (!res.ok) {
		return NextResponse.json(data, { status: res.status });
	}

	return NextResponse.json(data);
}
```
